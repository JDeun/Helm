# Helm 0.5.6

Helm 0.5.6 makes post-run memory decisions operationally explicit instead of leaving them half-hidden in task ledger prose.

## Highlights

- added typed memory operations for `write`, `promote`, `supersede`, `archive`, and `rollback`
- added crystallized session artifacts and `helm memory review-queue` so unresolved follow-up is inspectable
- surfaced memory operation counts, crystallized session counts, and memory review queue counts in `helm status` and `helm report`
- extended adaptive harness contracts with `route_decision`, `result_contract`, and `failure_downgrade`
- fixed state inspection so status/report follow the active workspace layout and read `.openclaw/` when Helm is operating over an OpenClaw-shaped workspace

## Why this release matters

Previous Helm releases could tell operators that durable capture mattered.
This release makes the next layer visible too:

- whether a run has been explicitly superseded
- whether a failed run was later closed through retry or archive logic
- whether a durable memory outcome has been crystallized into a compact artifact
- whether review-only work is still waiting in a visible queue
- whether Helm is reading the right state directory for the workspace shape you actually use

This is still a patch release.
The focus is not new runtimes.
The focus is reducing ambiguity in long-lived operator workflows.
