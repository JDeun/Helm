# Helm v0.6.2 — Model Health & Conversational Capture

**Date:** 2026-04-25

> Superseded by `v0.6.3`.
> `helm memory capture-chat` finalization planning had a release-following hotfix and the corrected tagged release is `v0.6.3`.

Helm 0.6.2 ports back the most practical runtime-state ideas from OpenClaw without dragging in OpenClaw-specific operational baggage. The result is a cleaner memory-capture core, a file-driven model-health subsystem, and a direct way to write durable capture records even when no profiled shell command ran.

## Highlights

### Model health

- Added `scripts/model_health_lib.py` and `scripts/model_health_probe.py`
- Added `helm health {probe,watch,select,state,launch}`
- Added `references/model_recovery_policy.json`
- Added model-health summary to `helm doctor`

This lets Helm persist recent model health, probe configured candidates, and select a fallback from files instead of relying on chat memory.

### Conversational durable capture

- Added `helm memory capture-chat`
- Added `scripts/conversational_memory_capture.py`
- Writes queued/running/final task-ledger events and a `memory_capture` payload without going through `run_with_profile.py`

This is intended for operator-driven or chat-driven work that still needs a task ledger trail and durable follow-up guidance.

### Memory capture refactor

- Extracted planning logic into `scripts/task_capture_core.py`
- Kept `scripts/memory_capture.py` as a compatibility wrapper
- Preserved the existing `build_memory_capture_plan()` entrypoint while adding touched-path support

This makes future memory-policy changes easier to isolate and test without coupling every planner decision to one large module.

### CLI hardening

- Fixed passthrough `--path` parsing in `helm.py` so nested subcommands like `helm memory capture-chat --path README.md` no longer lose their own arguments
- Extended `scripts/release_smoke.sh` to exercise the new `health` and `capture-chat` flows
- Updated the demo workspace to carry the new recovery policy template
