# Helm 0.5.3

Helm 0.5.3 turns browser-dependent work and blocked retrieval into first-class operational evidence instead of leaving those paths buried in prompt text or ad hoc notes.

## Highlights

- added `browser_work` and `retrieval_policy` evidence contracts to skill-owned adaptive harness manifests
- added retrieval escalation planning so blocked fetch paths can move through explicit stages and leave inspectable exit classifications
- added `backfill-evidence` so prior task-ledger runs can infer missing browser or retrieval evidence without rewriting original history
- expanded ledger reporting to show evidence coverage, retrieval exit classes, next-attempt stages, and required-evidence gaps by skill
- tightened manifest-quality auditing so conditional `when_any` triggers are checked for redundancy and overly broad hints

## Why this release matters

Helm already enforced execution profiles, context hydration, and finalization boundaries. This patch release closes another gap in long-lived workspaces:

- browser work should leave a reusable reason and next-action trail
- blocked retrieval should expose how the workflow escalated and where it stopped
- older runs should become more inspectable without pretending they were originally captured perfectly

The result is still a patch release, but it makes adaptive harness governance much easier to audit in real agent workspaces that mix local inspection, browser fallback, and external retrieval.
